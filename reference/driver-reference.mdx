OpenBeken uses a driver system to support the wide variety of hardware found inside Tuya-compatible devices. Each driver is a self-contained module that handles a specific chip, protocol, or feature. Drivers are started with `startDriver [DriverName]` — either from the web console, the **Config → Short startup command** field, or `autoexec.bat` on LittleFS. Many drivers start automatically when you configure the correct GPIO pin roles. Drivers are enabled at compile time per platform in `obk_config.h`; if a driver you need is not available on your platform, request it on the project's GitHub.

<Note>
  All driver descriptions are auto-generated from the firmware source code. For drivers marked as WIP, functionality may be incomplete or subject to change.
</Note>

## Protocol drivers

These drivers handle communication protocols between the WiFi module and external microcontrollers.

<AccordionGroup>
  <Accordion title="TuyaMCU — Tuya serial protocol">
    TuyaMCU handles the protocol between the WiFi module and the Tuya-compatible MCU found in most commercial devices. Communication uses the RX1/TX1 UART port. After starting this driver, use `linkTuyaMCUOutputToChannel` to map data point IDs (dpIDs) to OpenBeken channels.

    **Enable define:** `ENABLE_DRIVER_TUYAMCU`

    ```bash
    startDriver TuyaMCU
    # Change baud if needed (default is 9600)
    tuyaMcu_setBaudRate 115200
    # Map dpID 1 (bool) to channel 1
    linkTuyaMCUOutputToChannel 1 1 1
    ```

    See the [elektroda.com TuyaMCU dimmer example](https://www.elektroda.com/rtvforum/topic3929151.html) for a full walkthrough.
  </Accordion>

  <Accordion title="tmSensor — battery-powered TuyaMCU sensor">
    Used alongside TuyaMCU for low-power devices such as door sensors and humidity/temperature sensors. On boot, tmSensor requests a data update from the sensor MCU via TuyaMCU, reports it over MQTT, and then the MCU powers down the WiFi module and returns to sleep.

    **Enable define:** `ENABLE_DRIVER_TUYAMCU`

    ```bash
    startDriver TuyaMCU
    startDriver tmSensor
    ```
  </Accordion>

  <Accordion title="GirierMCU — Girier MCU protocol">
    Protocol driver for Girier-branded MCUs. Configuration is similar to TuyaMCU.

    **Enable define:** `ENABLE_DRIVER_GIRIERMCU`

    ```bash
    startDriver GirierMCU
    ```
  </Accordion>

  <Accordion title="UartTCP — UART to TCP bridge">
    Bridges the UART port to a TCP socket. Primarily used for WiFi Zigbee coordinators where a serial Zigbee module needs network access.

    **Enable define:** `ENABLE_DRIVER_UART_TCP`

    ```bash
    startDriver UartTCP
    ```
  </Accordion>
</AccordionGroup>

## Power metering drivers

These drivers communicate with energy metering chips to measure voltage, current, power, and energy consumption. After starting the driver, use `VoltageSet`, `CurrentSet`, and `PowerSet` to calibrate readings.

<AccordionGroup>
  <Accordion title="BL0942 — UART power metering">
    BL0942 communicates over UART (TX1/RX1 on BK chips). Default baud rate is 4800. You can use 9600 baud by passing it as an argument.

    **Enable define:** `ENABLE_DRIVER_BL0942`

    ```bash
    startDriver BL0942        # 4800 baud
    startDriver BL0942 9600   # 9600 baud
    VoltageSet 230.0
    CurrentSet 0.0
    PowerSet 0.0
    ```

    See the [LSPA9 teardown example](https://www.elektroda.com/rtvforum/topic3887748.html).
  </Accordion>

  <Accordion title="BL0942SPI — SPI power metering">
    SPI variant of BL0942, connected to the SPI1 port instead of UART. Used in some devices such as PZIOT-E01.

    **Enable define:** `ENABLE_DRIVER_BL0942SPI`

    ```bash
    startDriver BL0942SPI
    ```
  </Accordion>

  <Accordion title="BL0937 — pulse-output power metering">
    BL0937 uses a pulse-frequency output protocol rather than UART. Requires three GPIO pins: CF, CF1, and SEL. Setting all three BL0937 pin roles in the GPIO configuration causes the driver to start automatically.

    **Enable define:** `ENABLE_DRIVER_BL0937`

    Pin roles required: `BL0937_CF`, `BL0937_CF1`, `BL0937_SEL`
  </Accordion>

  <Accordion title="CSE7761 — dual-circuit energy metering">
    Single-phase multi-purpose energy metering chip with sigma-delta ADCs, SPI and UART interfaces. Driver is currently a work in progress.

    **Enable define:** `ENABLE_DRIVER_CSE7761`

    ```bash
    startDriver CSE7761
    ```
  </Accordion>

  <Accordion title="CSE7766 — UART power metering">
    CSE7766 communicates over UART (TX1/RX1), similar to BL0942.

    **Enable define:** `ENABLE_DRIVER_CSE7766`

    ```bash
    startDriver CSE7766
    ```
  </Accordion>

  <Accordion title="RN8209 — Zmai-90 power metering">
    Work-in-progress driver for the RN8209 chip found in some versions of the Zmai-90 smart socket.

    **Enable define:** `ENABLE_DRIVER_RN8209`

    ```bash
    startDriver RN8209
    ```
  </Accordion>

## LED drivers

These drivers control specialized LED chips found in smart bulbs and light strips. Most auto-start when you configure the required GPIO pin roles.

<AccordionGroup>
  <Accordion title="SM2135 — RGBCW I2C LED driver">
    SM2135 is a modified-I2C two-wire LED driver for RGBCW lights. The driver starts automatically when both `SM2135_DAT` and `SM2135_CLK` pin roles are set. Use `SM2135_Map` to correct the channel order for your device.

    **Enable define:** `ENABLE_DRIVER_LED`

    ```bash
    # Remap if colors are mixed
    SM2135_Map 0 2 1 3 4
    # Set max current codes
    SM2135_Current 2 4
    ```
  </Accordion>

  <Accordion title="SM2235 — RGBCW I2C LED driver (SM2235/SM2335/SM2185N)">
    Also covers SM2335 and SM2185N variants. Auto-starts when `SM2235_DAT` and `SM2235_CLK` pin roles are set.

    **Enable define:** `ENABLE_DRIVER_LED`

    ```bash
    SM2235_Map 0 1 2 3 4
    SM2235_Current 20
    ```
  </Accordion>

  <Accordion title="BP5758D — RGBCW I2C LED driver">
    Used in some BL602 and Sonoff bulbs. Auto-starts when `BP5758D_DAT` and `BP5758D_CLK` are set.

    **Enable define:** `ENABLE_DRIVER_LED`

    ```bash
    BP5758D_Map 0 1 2 3 4
    BP5758D_Current 10 10
    ```
  </Accordion>

  <Accordion title="BP1658CJ — RGBCW I2C LED driver">
    Similar to BP5758D. Auto-starts with `BP1658CJ_DAT` and `BP1658CJ_CLK` pin roles.

    **Enable define:** `ENABLE_DRIVER_LED`

    ```bash
    BP1658CJ_Map 0 1 2 3 4
    BP1658CJ_Current 10 10
    ```
  </Accordion>

  <Accordion title="KP18058 — I2C LED driver (KP18058/KP18068)">
    Supports both KP18058 and KP18068 chips. Auto-starts with `KP18058_CLK` and `KP18058_DAT` roles.

    **Enable define:** `ENABLE_DRIVER_KP18058`

    ```bash
    KP18058_Map 0 1 2 3 4
    # Values 0-31 for RGB, 0-31 for CW
    KP18058_Current 14 30
    ```
  </Accordion>

  <Accordion title="PixelAnim — WS2812B animations">
    Provides a set of built-in animations for WS2812B addressable LED strips. Use `Anim [index]` to start an animation and `AnimSpeed [interval]` to control speed.

    **Enable define:** `ENABLE_DRIVER_PIXELANIM`

    ```bash
    startDriver PixelAnim
    Anim 3
    AnimSpeed 50
    ```
  </Accordion>

  <Accordion title="SM16703P — SPI addressable LED driver (WS2812B-compatible)">
    Controls individually addressable LEDs via hardware SPI. The data line connects to P16 (MOSI). Also used for WS2812B and SM15155E strips.

    **Enable define:** `ENABLE_DRIVER_SM16703P`

    Pin role: `SM16703P_DIN`

    ```bash
    startDriver SM16703P
    SM16703P_Init 60 RGB
    SM16703P_SetPixel all 255 0 0    # all red
    SM16703P_SetPixel 0 0 255 0      # first pixel green
    SM16703P_Start
    ```
  </Accordion>

  <Accordion title="SM15155E — single-wire 5-channel LED driver">
    WS2812B-like single-wire controller for 5-channel LED output. Uses P16 (SPI out) on Beken.

    **Enable define:** `ENABLE_DRIVER_SM15155E`

    Pin role: `SM16703P_DIN` (same role as SM16703P)

    ```bash
    startDriver SM15155E
    ```
  </Accordion>

  <Accordion title="PWMToggler — PWM on/off with value memory">
    An abstraction layer on top of raw PWM channels that allows toggling the PWM output on and off while preserving its value. Useful for custom lights or laser devices where you need power control separate from intensity.

    **Enable define:** `PLATFORM_BEKEN` or `WINDOWS`

    ```bash
    startDriver PWMToggler
    toggler_channel0 3   # link toggler 0 to channel 3
    toggler_enable0 1
    toggler_set0 75
    ```
  </Accordion>

  <Accordion title="Drawers — WS2812B smart drawers">
    WS2812B driver wrapper with a REST API for the smart drawers project.

    **Enable define:** `ENABLE_DRIVER_DRAWERS`
  </Accordion>
</AccordionGroup>

    ```bash

    scanI2C Soft    # scan for connected devices
    ```

## Protocol and integration drivers

<AccordionGroup>
  <Accordion title="DDP — distributed display protocol">
    DDP is a UDP-based LED control protocol supported by xLights, Falcon Player, and other sequencing software. Allows external apps to control OpenBeken LED strips in real time.

    **Enable define:** `ENABLE_DRIVER_DDP`

    ```bash
    startDriver DDP
    ```
  </Accordion>

  <Accordion title="DDPSend — send DDP to another device">
    Sends DDP packets to another device. See the [DDP topic](https://www.elektroda.com/rtvforum/topic4040325.html) for examples.

    **Enable define:** `ENABLE_DRIVER_DDPSEND`

    ```bash
    startDriver DDPSend
    DDP_Send [ip] [host] [pixelsize] [delay] [data]
    ```
  </Accordion>

  <Accordion title="SSDP — network discovery">
    SSDP (Simple Service Discovery Protocol) allows OpenBeken devices to appear in the Windows Network section and enables Alexa discovery. Must be running before Wemo or Hue.

    **Enable define:** `ENABLE_DRIVER_SSDP`

    ```bash
    startDriver SSDP
    # List discovered OBK devices on the network
    obkDeviceList
    ```
  </Accordion>

  <Accordion title="Wemo — Alexa Wemo emulation">
    Emulates a Belkin Wemo device for Alexa voice control. Requires SSDP to be running.

    **Enable define:** `ENABLE_DRIVER_WEMO`

    ```bash
    backlog startDriver SSDP; startDriver Wemo
    ```
  </Accordion>

  <Accordion title="Hue — Alexa Hue emulation">
    Emulates a Philips Hue bridge for Alexa voice control. Requires SSDP.

    **Enable define:** `ENABLE_DRIVER_HUE`

    ```bash
    backlog startDriver SSDP; startDriver Hue
    ```
  </Accordion>

  <Accordion title="DGR — Tasmota Device Groups">
    Implements the Tasmota Device Groups protocol for synchronized control of multiple devices without a central MQTT broker.

    **Enable define:** `ENABLE_TASMOTADEVICEGROUPS`

    ```bash

    DGR_SendBrightness MyGroup 80
    DGR_SendRGBCW MyGroup FF8800
    ```

  <Accordion title="IR / IR2 — infrared remote control">

    - **IR2** — simpler IR driver for sending captures from Flipper Zero.

## Utility drivers

<AccordionGroup>
  <Accordion title="NTP — network time synchronization">
    Fetches the current time from an NTP server. Required for `addClockEvent`, sunrise/sunset variables, and timestamp-based logging. Without NTP, there is no correct datetime.

    **Enable define:** `ENABLE_NTP`

    ```bash
    startDriver NTP
    ntp_setServer pool.ntp.org
    ntp_timeZoneOfs 1        # UTC+1
    ```
  </Accordion>

  <Accordion title="DS3231 — hardware RTC">
    Driver for the DS3231 real-time clock over I2C. Can sync the device clock from the RTC on startup or keep the device clock in sync from the RTC.

    **Enable define:** `ENABLE_DRIVER_DS3231`

    ```bash
    # Start with CLK pin 5, DATA pin 6, sync mode 1 (set device clock from RTC on start)
    startDriver DS3231 5 6 1
    DS3231_GetTime
    DS3231_SetTime 19 50 01 13 8 25
    ```
  </Accordion>

  <Accordion title="HTTPButtons — custom web interface buttons">
    Creates custom, scriptable buttons on the device's main web page. Define buttons and their commands in `autoexec.bat`.

    **Enable define:** `ENABLE_DRIVER_HTTPBUTTONS`

    ```bash
    startDriver HTTPButtons
    setButtonLabel 0 "Scene 1"
    setButtonCommand 0 "backlog led_basecolor_rgb FF0000; led_dimmer 80"
    setButtonColor 0 red
    setButtonEnabled 0 1
    ```
  </Accordion>

  <Accordion title="Charts — on-device data charting">
    Creates customizable charts displayed directly on the device's web interface. Data is logged via `chart_add` or `chart_addNow` commands.

    **Enable define:** `ENABLE_DRIVER_CHARTS`

    ```bash
    startDriver Charts
    chart_create 120 2 1
    chart_setAxis 0 Time 0 Time
    chart_setVar 0 Temperature 0
    chart_setVar 1 Humidity 0
    chart_addNow $CH1 $CH2
    ```
  </Accordion>

  <Accordion title="Widget — custom HTML widgets">
    Loads custom HTML snippets from LittleFS files and displays them on the main OBK web page. Snippets can use the OBK REST API.

    **Enable define:** `ENABLE_DRIVER_WIDGET`

    ```bash
    startDriver Widget
    widget_create 0 1 mywidget.html
    ```
  </Accordion>

  <Accordion title="ADCButton — multiple buttons on one ADC pin">
    Connects multiple buttons to a single ADC pin using a resistor ladder. Each button has a different resistor value producing a unique voltage level. Use `AB_Map` to configure margins.

    **Enable define:** `ENABLE_DRIVER_ADCBUTTON`

    Pin role: `ADC_Button`

    ```bash
    startDriver ADCButton
    AB_Map 4    # 4 margins = 5 possible button values
    ```
  </Accordion>

  <Accordion title="DoorSensor — deep sleep door/contact sensor">
    Uses BK GPIO deep sleep for ultra-low-power door or contact sensors. The device wakes on a GPIO event, reports the change over MQTT, and goes back to sleep. Use the `DoorSensorWithDeepSleep` pin role for auto-start.

    **Enable define:** `PLATFORM_BEKEN` or `WINDOWS`

    ```bash
    startDriver DoorSensor
    DSTime 30         # stay awake 30s after last event
    DSEdge 2          # auto-detect rising/falling based on current state
    ```
  </Accordion>

  <Accordion title="ADCSmoother — staircase switch synchronization">
    Used for 3-way staircase switches synchronized over an extra wire via ADC voltage levels.

    **Enable define:** `ENABLE_DRIVER_ADCSMOOTHER`

    ```bash

    ```
  </Accordion>

  </Accordion>

<Tip>
